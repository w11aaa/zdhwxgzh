## 一、项目背景与目标

### 1.1 业务痛点

考公、事业单位、教师、国企等招聘公告分散在 20+ 个网站（国家公务员局、各省人事考试网、事业单位招聘平台、高校就业网等），存在以下核心问题：

| 痛点 | 具体表现 |
|------|----------|
| 信息分散 | 同一考试类型的公告分布在多个网站，格式不统一 |
| 时效敏感 | 报名截止日期严格，错过一天可能错过整个招聘季 |
| 附件复杂 | 岗位表散落在 .xlsx/.docx/.pdf 中，人工下载解析效率低 |
| 内容合规 | 公告属于事实敏感内容，不能凭空补充或改写 |
| 重复劳动 | 运营人员每天要花 2-3 小时查找、整理、排版 |

### 1.2 建设目标

构建一套 **Multi-Agent 智能协作系统**，实现从数据采集 → 结构化入库 → 智能检索 → 内容生成 → 质量审核 → 自动发布的全链路自动化，将人工 3 小时的工作压缩到 5 分钟内完成。

### 2.2 技术选型

| 层级 | 技术 | 选型理由 |
|------|------|----------|
| **Web 框架** | Python http.server (标准库) | 零外部依赖，单文件部署，适合个人项目快速验证 |
| **数据库** | SQLite (WAL 模式) | 轻量嵌入式，无需运维，15 张表支撑全部业务 |
| **向量数据库** | FAISS (Facebook) | 高性能向量检索，纯 CPU 运行，支持 Inner Product 相似度 |
| **Embedding** | BAAI/bge-small-zh-v1.5 | 中文语义理解 SOTA，512 维，本地运行不依赖外部 API |
| **LLM** | DeepSeek API (v4-flash) | 性价比最高的中文大模型，支持 Chat Completions |
| **图算法** | SQLite + NetworkX | 用 SQLite 存储图结构（节点表+边表），轻量无部署成本 |
| **浏览器自动化** | Playwright | 支持 Chromium，用于公考雷达页面抓取和原文搜索 |
| **流式推送** | SSE (Server-Sent Events) | 比 WebSocket 更简单，单向推送足够对话场景 |
| **任务调度** | Python threading + subprocess | 轻量异步执行 Agent 任务，无需 Celery 等重量级组件 |

## 三、核心技术方案

### 3.1 RAG 语义检索引擎

**问题**：用户输入"海淀区有没有计算机类的事业编"，传统基于 SQL LIKE 的检索无法处理"海淀区"→"北京海淀"、"计算机类"→正文匹配、"事业编"→"事业单位"的语义转换。

**方案**：混合检索（Hybrid Search）架构

```
用户查询: "海淀区有没有计算机类的事业编"
              │
    ┌─────────┴─────────┐
    │                   │
    ▼                   ▼
┌──────────┐      ┌──────────┐
│ 向量语义路 │      │ 关键词精确路│
│ Embedding │      │ SQL FTS  │
│ + FAISS   │      │ + LIKE   │
│ top-15    │      │ top-15   │
└────┬─────┘      └────┬─────┘
     │                 │
     └───────┬─────────┘
             ▼
    ┌──────────────┐
    │ RRF 融合排序  │  ← Reciprocal Rank Fusion
    │ score = Σ 1/(k+rank+1) │  两路结果加权合并
    └──────┬───────┘
           ▼
      Top-5 结果
```

**关键参数**：

- Embedding 模型: `BAAI/bge-small-zh-v1.5` (512 维)
- 索引结构: `faiss.IndexFlatIP` (Inner Product，适用于归一化向量的余弦相似度)
- RRF 参数: `k=60`
- 索引大小: 737 条 × 512 维 ≈ 1.5MB
- 检索耗时: <50ms (FAISS 搜索) + 加载模型 4.6s (首次)

### 3.2 知识图谱引擎

**问题**：用户查看一条公告时，系统应能推荐"同一地区相同考试类型的其他公告"或"同一单位的历史公告"。

**方案**：轻量图数据库设计

```
节点类型 (5种):
  Region(36) ──BELONGS_TO──▶ Announcement(737) ◀──PUBLISHES── Organization(737)
                                       │
                                  HAS_CATEGORY
                                       │
                                       ▼
                                  ExamType(14)

关系边 (3种 + 相似边):
  - BELONGS_TO: 公告 → 地区
  - PUBLISHES: 单位 → 公告
  - HAS_CATEGORY: 公告 → 考试类型
  - SIMILAR_TO: 公告 ↔ 公告 (共享地区/类型)
```

**存储方案**：在 SQLite 中用节点表 + 边表模拟图结构

```sql
kg_nodes (node_id, node_type, label, properties)
kg_edges (edge_id, source_id, target_id, relation, weight)
```

**关联推荐算法**：同一个地区的同类型公告 → 同一单位的其他公告 → 相似公告（共享地区或类型的公告，Jaccard 相似度加权）

### 3.3 Multi-Agent 协作

**问题**：单一 Agent 难以处理从采集到发布的复杂长链路，需要分工协作。

**方案**：专业 Agent + 编排器模式

```
MultiAgentOrchestrator
    │
    ├── CrawlerAgent   (role: CRAWLER)
    │   tools: [crawl_fenbi, crawl_gongkaoleida, search_event_origin]
    │   tasks: [check_origin, search_origin, get_event_info, collect_new]
    │
    ├── EditorAgent    (role: EDITOR)
    │   tools: [wechat_article_generate, attachment_image]
    │   tasks: [generate_draft, generate_title_only]
    │
    ├── QAAgent        (role: QA)
    │   tools: [quality_check, fact_consistency_check]
    │   tasks: [quality_check, fact_check, diagnose_error]
    │
    └── PublisherAgent (role: PUBLISHER)
        tools: [wechat_draft_submit, check_wechat_token]
        tasks: [submit_draft, check_token, export_preview]

通信协议: AgentMessage
  { msg_id, from_role, to_role, task_type, payload,
    status, observation, reasoning, error, elapsed_ms }
```

### 3.4 对话系统三层路由

```
用户消息
    │
    ├── L1: 关键词匹配 (<10ms)
    │   ├─ 地区+类型识别 (100+ 映射规则)
    │   ├─ SQL 精确查询 → 立即返回
    │   └─ 帮助/推荐等固定回复
    │
    ├── L2: RAG + DeepSeek 增强 (1-5s)
    │   ├─ 向量语义检索公告正文
    │   ├─ 知识图谱关联推荐
    │   ├─ 检索结果注入 System Prompt
    │   └─ DeepSeek 基于原文生成回答
    │
    └── L3: 兜底 (<1s)
        └─ 固定回复 + 引导用户使用正确格式
```

### 3.5 流式推送 (SSE)

采用 Server-Sent Events 实现对话的逐字推送：

```
Server                          Browser
  │                               │
  │  GET /api/chat/stream?msg=xx  │
  │◄──────────────────────────────│
  │                               │
  │  data: {"type":"thinking"}    │
  │──────────────────────────────►│ → [用户看到: 🔍 分析中...]
  │                               │
  │  data: {"type":"tool_call",   │
  │    "tool":"rag_search",       │
  │    "count":5}                 │
  │──────────────────────────────►│ → [用户看到: RAG search 5 results]
  │                               │
  │  data: {"type":"reply_chunk", │
  │    "content":"为您找到"}      │
  │──────────────────────────────►│ → [用户看到: 为您找到...]
  │                               │
  │  data: {"type":"done"}        │
  │──────────────────────────────►│ → [流结束]
```

conda activate dome
cd /d F:\Automated_operation
python -B -m kaoyan_collector.ui_app --host 127.0.0.1 --port 7860
