## 一、整体数据流设计

text

```
[政府/高校/国企官网] + [第三方聚合站] + [公众号/订阅]
         ↓ (定时抓取，频率按源分级)
   原始HTML/JSON → 规则引擎去噪 → 清洗为结构化字段
         ↓
       MySQL/PostgreSQL 入库（带状态标记）
         ↓
   生成引擎：选取“今日/本周可报” → 套模板 → 文+图
         ↓
    自动/人工微调后发布到小红书
         ↓
   (可选) 跟踪已发布内容的报名截止，自动发“即将关闭”提醒
```

## 二、数据源分级与抓取策略

考公信息高度分散，需要分三级来源，采用不同抓取频次。

### 1. 一级源：国家级/省级官方公告（核心，更新不快但权威）

- **国家公务员局**（国考）：公告、职位表、报名入口
- **各省人事考试网**（省考）：如“浙江省人事考试网”，含公告+岗位表
- **事业单位招聘**：如“中央和国家机关所属事业单位公开招聘服务平台”、“全国事业单位招聘网”
- **抓取频率**：每日1次即可，一般在工作日早9-10点检查更新

### 2. 二级源：垂直聚合站（结构化好，易抓取）

- **公考雷达**（有Web版，岗位表结构化程度高）
- **中公/华图日历页**（他们有汇总的“近期报名”页）
- **高校就业指导中心**（选调生、国企校招，如“北大就业网”）
- **抓取频率**：每2-4小时一次，用移动端UA或API抓取（部分有JSON）

### 3. 三级源：微信公众号/小程序（时效最快，但需特殊处理）

- **各省考试院/人社厅官方公众号**
- **国企招聘公众号**（如“国资小新”、“XX集团招聘”）
- **获取方式**：可用搜狗微信搜索接口抓取最新文章标题和时间，或订阅RSS化服务；更简单的是抓取“微信公众平台公开的Sitemap”或使用第三方舆情工具（如新知魔方、WeTool）导出。

> **反爬应对**：人事网多有Cloudflare或反爬，可用Headless Browser(Playwright) + IP池。二级源相对宽松，优先从二级源抓取，一级源做核验。

## 三、去噪与结构化规则设计

爬回来的内容五花八门，你的“规则去噪”模块需要输出统一的**招聘事件卡片**。我设计一套字段与规则：

### 必备字段（数据库表字段）

| 字段                    | 说明          | 清洗规则                                                   |
| :---------------------- | :------------ | :--------------------------------------------------------- |
| `title`                 | 标准化标题    | 从原文提取，补充【省份】【单位】【类型】                   |
| `region`                | 省份/城市     | 从来源站或标题地理词库映射                                 |
| `category`              | 类型          | 规则分类：国考/省考/事业单位/选调/国企/军队文职/三支一扶等 |
| `org_name`              | 招聘单位      | 正则提取“XX局”“XX公司”                                     |
| `job_count`             | 招聘人数      | 提取数字，无则空                                           |
| `qualification`         | 学历要求      | 提取“本科及以上”“硕士”等，可为多选                         |
| `major_requirements`    | 专业要求      | 特别提取“计算机”“不限”等关键词，用于筛选                   |
| `registration_start`    | 报名开始日期  | 标准化为YYYY-MM-DD                                         |
| `registration_deadline` | 报名截止日期  | **最核心字段**，必须提取准确                               |
| `exam_date`             | 笔试/面试时间 | 提取，可能为空                                             |
| `source_url`            | 原文链接      | 保留作为引用                                               |
| `raw_text`              | 原始文本      | 备用，调试用                                               |
| `hash_id`               | 内容指纹      | MD5(title+deadline+org_name)做去重                         |
| `status`                | 状态          | 自动标记：即将开始/正在报名/报名截止/已结束                |

### 去噪与提取关键技术

- **日期提取**：用正则匹配各种中文日期格式（“2026年6月10日9:00至6月20日17:00”），并转换成标准区间。若发现截止日期已过，状态直接置为“已结束”不发布。
- **标题标准化**：原始标题常缺失地域，如“2026年度考试录用公务员公告”，你需要根据来源站追加【国考】【北京】等前缀，用规则或规则树自动补全。
- **重复过滤**：同一公告会被多个源抓取，用`hash_id`去重，保留最早抓取的那条或来源最权威的。
- **过滤无关内容**：利用关键词排除“结果公示”“面试递补”“体检通知”，只留下招考公告、报名通知、岗位表等。

### 状态流转换逻辑

text

```
抓取入库时判断：
if 报名开始日期 > 今天: status='即将开始'
elif 今天 ≤ 报名截止日期: status='正在报名'  （加入“今日可报”内容池）
else: status='已结束'（归档，不发布）
```



每天凌晨统一跑一遍状态更新，将过期内容的`status`改为“已结束”。

## 四、内容生成模板设计

你的系统从库中筛选 `status = '正在报名'` 且 `registration_deadline` 在近3-7天内的记录，按热度/报名人数/地区组合，生成帖子。

### 标题公式（可直接代码生成）

- **紧急提醒型**：
  `【#{region} #{org_name}】#{category}报名#{deadline_diff}天后截止！招#{job_count}人，#{qualification}可报`
  （deadline_diff为距今天数）
- **整合清单型**：
  `本周必看！这#{count}个#{category}岗位即将关闭报名通道，含#{region}`
- **低门槛型**：
  `不限专业！#{region}这些事业单位招人中，本科起报`
- **高薪编制型**：
  `铁饭碗！#{org_name}招#{job_count}人，有编制，计算机类可报`

系统根据字段动态组合，A/B测试哪种打开率高。

### 文案模板（正文结构）

text

```
🔥 报名倒计时：截止到#{registration_deadline}
👀 招聘单位：#{org_name}
📌 岗位性质：#{category}（有编制/合同制自动判断）
📊 招聘人数：#{job_count}人
🎓 学历要求：#{qualification}
💻 专业要求：#{major_requirements}
📅 报名时间：#{registration_start} 至 #{registration_deadline}
⏰ 笔试时间：#{exam_date ? exam_date : '详见公告'}

🔗 报名入口及职位表：见评论区（避免引流违规，可用“口令”或引导私信，或直接放官网截图中的二维码）

#考公上岸 #公务员考试 #事业单位招聘 #体制内打工人 #信息差
```



文案最后可加固定话术：“关注我，每天更新最新体制内招聘，打破信息差。”

### 图片模板（自动化制图）

可用 Python 的 Pillow/Playwright截图 或 前端Canvas生成卡片。

- **单岗位卡片**：深蓝背景+黄色强调数字（截止天数、招聘人数），布局像日历提醒。
- **合集信息表**：表格形式，列出“省份/单位/截止时间/人数”，适合整合帖。
- **可准备3套模板**：通用公告卡、国企校招卡、紧急提醒卡，按`category`自动选择。

图片下方留白区可放小红书号或引导关注图标。

## 五、发布节奏与流量优化

- **发布频率**：每日早9点、中午12点、晚7点各1条（对应考公人群的通勤/午休/下班后）。
- **内容组合**：
  - 单日可安排：一条“今日可报合集”（整合多个岗位，收藏率高），一条“紧急提醒”（某个热门岗位马上截止），一条“捡漏冷门”（竞争小的编制）。
- **评论区维护**：
  - 系统可自动抢首评：“需要职位表/报名入口的扣1，我发你”，通过私信（防违规）或企微导流建私域。
  - 后期可接入简单关键词回复Bot，用户评论“省份”自动回复该省报名链接合集。
- **流量节点**：国考/省考公告季、报名最后一天、准考证打印日，需加大发布密度。

## 六、技术难点与应对

| 难点                 | 应对方案                                                     |
| :------------------- | :----------------------------------------------------------- |
| 日期提取错误         | 在规则之后加一层置信度判断，若无法准确解析日期，标记为“需人工审核”暂不发 |
| 同一公告多源时间冲突 | 以官方一级源为准，聚合时设定权威优先级                       |
| 反爬导致断流         | 预备多个二级源交替抓取，建异常监控，抓取失败自动切换备用渠道或暂存重试 |
| 账号被判定营销       | 文案避免“链接”“加微信”，用截图引导；账号初期先做纯信息分享，积累权重 |

------

## 七、建议的MVP起步路径

1. **只抓5-8个高质量二级源**（如公考雷达、全国事业单位招聘网、各省人事考试网TOP5省份），先跑通“抓取→去重→提取截止日期→生成单张卡片”的最小闭环。
2. 生成内容先发到内部群审核，确认准确率>95%再自动发。
3. 稳定后逐步扩展省份和考试类型，加入“选调”“军队文职”“央企校招”等更细类目。
